# AVLB AI — Adaptive Validator Load Balancer

> Интеллектуальный маршрутизатор транзакций для Solana, который анализирует состояние сети в реальном времени и направляет транзакции через оптимальный валидатор.

---

## Проблема

В сети Solana три фундаментальные проблемы:

**1. Слепая маршрутизация** — транзакции отправляются без учёта реального состояния валидаторов, случайно, вслепую.

**2. Устаревшие метрики** — данные о нагрузке узлов распространяются через gossip-протокол и к моменту принятия решения уже устарели, особенно между регионами (Азия → Европа → США).

**3. Фиксированная логика** — существующие решения используют статичные формулы. Они не знают, что во время NFT-минта latency важнее CPU, а при CRITICAL-перегрузке нужна совсем другая стратегия.

**Результат:** очереди растут, задержки увеличиваются, транзакции теряются.

---

## Решение

AVLB AI решает все три проблемы одновременно:

| Проблема | Решение |
|---|---|
| Слепая маршрутизация | Adaptive Scoring Engine выбирает лучший узел |
| Устаревшие метрики | Географически распределённые Relay Nodes |
| Фиксированная логика | Динамические веса под контекст сети |

---

## Архитектура

```
[Solana Validators]
        ↓ метрики
[Relay Node - Азия] ←——→ [Relay Node - Европа] ←——→ [Relay Node - США]
        ↓
[Adaptive Scoring Engine (MLP Neural Network)]
  → определяет режим: NORMAL / HIGH_LOAD / CRITICAL
  → динамически меняет веса метрик
  → выбирает лучший валидатор
        ↓
[AVLB Router]
  → записывает решение on-chain (Solana Smart Contract)
  → отправляет транзакцию через оптимальный узел
        ↓
[Solana Devnet]
```

## Как это работает

### Adaptive Scoring Engine

Вместо фиксированной формулы:
```
❌ score = queue + latency + cpu  (одинаково всегда)
```

AVLB AI адаптирует веса под контекст:

| Режим | Когда | Приоритет |
|---|---|---|
| `NORMAL` | Обычная активность | Баланс всех метрик |
| `HIGH_LOAD` | NFT минт / DEX всплеск | Latency > CPU |
| `CRITICAL` | Перегрузка сети | Уход от занятых узлов |

**Метрики которые анализирует система:**
- `latency_ms` — задержка узла
- `load_percent` — загрузка CPU
- `success_rate` — успешность транзакций
- `mempool_size` — размер очереди
- `priority_fee` — приоритетная комиссия
- `sync_diff` — отставание от актуального блока сети

### Neural Network

Система использует **MLPRegressor** (многослойный перцептрон) с архитектурой:
- Два скрытых слоя (64, 32 нейрона)
- Функция активации: ReLU
- Нормализация данных: StandardScaler
- Обучение на реальных метриках Solana (история 100 состояний каждого валидатора через Redis)

Это позволяет системе находить **нелинейные зависимости** в поведении валидаторов — то, что фиксированная формула никогда не обнаружит.

### Relay Nodes

Relay Node — географически распределённый узел с двойной передачей метрик:

```
Relay получил метрику
        ↓
  ┌─────┴──────┐
  ↓            ↓
Локально    Клонам в других
(ближние    регионах
валидаторы) мгновенно
```

**Что это даёт:**
- Метрики не устаревают пока добираются через полмира
- Каждый регион видит актуальное состояние всей сети
- Нет единой точки отказа

### Smart Contract (On-Chain Proof)

Каждое решение AI **записывается в блокчейн Solana**. Это делает систему прозрачной и верифицируемой.

Что фиксируется on-chain:
- Выбранный валидатор
- Режим сети в момент решения
- Score валидатора
- Timestamp

```
AI принял решение → Записал в Solana Devnet → Доступно в Explorer
```

## Сравнение с рынком

| | Helius | Triton | Jito | **AVLB AI** |
|---|---|---|---|---|
| Что делает | Даёт данные | Даёт скорость | MEV маршруты | **Выбирает куда слать** |
| Адаптация к сети | ❌ | ❌ | ❌ | ✅ |
| Режимы нагрузки | ❌ | ❌ | ❌ | ✅ |
| On-chain proof | ❌ | ❌ | ❌ | ✅ |
| Работает поверх других | ❌ | ❌ | ❌ | ✅ |

**Ключевое отличие:** Helius, Triton и Jito — это инфраструктура (трубы). AVLB AI — это мозг, который решает через какую трубу отправить транзакцию прямо сейчас.

AVLB не конкурирует с ними — он работает **поверх** них как интеллектуальный слой.
